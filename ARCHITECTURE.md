## 1. 시스템 개요

### 1.1 목적

본 시스템은 한국투자증권 Open API를 활용하여 국내 주식 및 ETF(금, 은, 채권 등) 시장에서 자동으로 매매와 자산 리밸런싱을 수행하는 알고리즘 트레이딩 플랫폼이다. 시장 데이터 수집, 전략 기반 신호 생성, 주문 실행, 리스크 관리, 뉴스 분석, 장세별 자산군 리밸런싱, 성과 평가 및 전략 자동 개선을 통합적으로 처리한다.

### 1.2 시스템 환경

| 항목 | 사양 |
|------|------|
| 서버 | Lenovo IdeaPad 1 (Ubuntu 24.04 LTS) — 로컬 상시 가동 |
| 언어 | Python 3.11+ |
| 증권사 API | 한국투자증권 Open API (REST + WebSocket) |
| 데이터베이스 | SQLite (경량, 무료) |
| 스케줄러 | APScheduler + crontab |
| 알림/보고 | Discord Bot (무료) |
| AI 엔진 | ChatGPT Codex 5.3 (OAuth, ChatGPT Plus $20/월 구독료만 발생) |
| 보고 UI | Streamlit 대시보드 (무료, 로컬 브라우저) |
| 버전관리/배포 | GitHub + Claude Code → git clone 으로 로컬 배포 |
| 비용 구조 | ChatGPT Plus 구독료($20/월) 외 전부 무료 |

### 1.3 API 호출 제한 및 대응

| 구분 | 제한 | 대응 전략 |
|------|------|-----------|
| REST API (실전) | 초당 20건 | 쓰로틀링 적용, 초당 15건 이하로 제한 |
| REST API (모의) | 초당 5건 | 개발/테스트 시 무조건 모의투자 우선 |
| 웹소켓 실시간 | 세션당 41종목 | 관심종목 우선순위 관리 |
| 액세스 토큰 | 24시간 유효 | 자동 갱신 모듈 구현 |

시스템은 슬라이딩 윈도우 방식의 유량 제한을 고려하여, REST API 호출 시 초당 15건 이하로 안전 마진을 두고 쓰로틀링을 적용한다. 분봉 데이터는 REST로, 실시간 체결가/호가는 웹소켓으로 수신하는 하이브리드 구조를 채택한다.

## 3. 데이터 수집 모듈

### 3.1 주식 데이터 수집 (data_collector.py)

주식 데이터는 두 가지 채널을 병행하여 수집한다.

#### 3.1.1 REST API 수집

| 수집 항목 | 주기 | API 엔드포인트 | 비고 |
|-----------|------|---------------|------|
| 일봉 데이터 | 08:50 (1일 1회) | /uapi/domestic-stock/v1/quotations/inquire-daily-itemchartprice | 전략 신호 생성의 핵심 데이터. 2회 호출로 100건 확보 |
| KOSPI 업종지수 일봉 | 08:50 (1일 1회) | /uapi/domestic-stock/v1/quotations/inquire-daily-indexchartprice | 장세 판단용. FID_COND_MRKT_DIV_CODE="U" |
| 현재가 조회 | 매 5분 (트레이딩 사이클) | /uapi/domestic-stock/v1/quotations/inquire-price | 일봉에 당일 바 추가용 (시가/고가/저가/현재가) |
| 투자자별 매매동향 | 매 30분 | HTS 크롤링 또는 API | 개인/외국인/기관 순매수 |
| 잔고 조회 | 매 10분 | /uapi/domestic-stock/v1/trading/inquire-balance | 보유 종목, 평가손익 |

#### 3.1.2 WebSocket 실시간 수신

웹소켓은 실시간 체결가와 호가 데이터를 수신한다. 세션당 최대 41개 종목을 구독할 수 있으므로, 관심종목 우선순위에 따라 동적으로 구독을 관리한다.

| 구독 타입 | 용도 | 우선순위 |
|-----------|------|----------|
| 실시간 체결가 (H0STCNT0) | 현재 체결 가격/량 모니터링 | 보유 종목 > 매수 후보 > 관심종목 |
| 실시간 호가 (H0STASP0) | 호가 창 변화 감지 | 보유 종목 우선 |

### 3.2 뉴스 데이터 수집 (news_collector.py)

뉴스는 고정 시간에 3회 수집하며, 다양한 소스를 활용한다.

| 수집 시간 | 소스 | 방법 | 수집 대상 |
|-----------|------|------|-----------|
| 아침 08:00 | 네이버 금융뉴스, 한경비즈니스 | 웹 크롤링 (BeautifulSoup) | 전일 약정, 해외시장 동향, 금리/환율 |
| 점심 12:00 | KRX 공시, 연합뉴스 | API + 크롤링 | 오전 장 요약, 기업 공시, 당일 이슈 |
| 오후 14:00 | 증권사 리포트, 경제지표 | API + 크롤링 | 오후 장 전망, 증권사 목표가 |

#### 3.2.1 크롤링 CSS 셀렉터

| 소스 | URL | 셀렉터 | 비고 |
|------|-----|--------|------|
| 네이버금융 | `finance.naver.com/news/mainnews.naver` | `.articleSubject a` | 2026-03-30 수정 |
| 한국경제 | `www.hankyung.com/economy` | `.news-tit a` | 2026-03-30 수정 |

> 외부 사이트 구조 변경에 취약하므로 매주 일요일 `scripts/check_news_selectors.py`로 자동 점검한다.

#### 3.2.2 뉴스 처리 파이프라인

1. 웹 크롤링/API로 원문 수집
2. 중복 제거 (제목 해시값 기반)
3. AI API로 감성 분석 (긍정/부정/중립, 점수 -1.0 ~ +1.0)
4. 관련 종목 매핑 (종목명 추출)
5. DB 저장 및 전략 엔진에 전달

AI 감성 분석은 ChatGPT Codex 5.3를 OAuth 방식으로 연동하여 뉴스 본문에서 주가에 미치는 영향도를 점수화한다. ChatGPT Plus 구독 범위 내에서 처리되므로 추가 API 비용이 발생하지 않는다.

## 4. 종목 선정 모듈

### 4.1 종목 선정 구조

종목 선정은 고정 유니버스와 동적 스크리닝을 혼합하여 운영한다. ETF 자산배분 종목은 고정 유니버스로, 개별 주식은 매일 아침 전체 시장에서 조건에 맞는 종목을 자동 필터링한다. 시스템이 관리하는 종목은 세 계층으로 구분된다:

| 계층 | 설명 | 변경 주기 |
|------|------|-----------|
| 고정 유니버스 | ETF 자산배분 종목 + 수동 등록 관심종목 | 수동 관리 |
| 스크리닝 후보군 | 매일 아침 자동 필터링된 종목 (30~50개) | 매일 08:30 |
| 보유 종목 (필수) | 현재 보유 중인 모든 종목 — 절대 제외 불가 | 실시간 자동 |

### 4.2 동적 스크리닝 (2단계 필터링)

매일 08:30에 2단계 필터링으로 후보 종목을 선정한다. API 초당 20건 제한을 고려하여 1차에서 대폭 축소 후 2차에서 정밀 필터링한다.

**1차 필터: 기본 요건** (현재가 API + 일봉 API)

| 필터 | 조건 | 목적 |
|------|------|------|
| 시가총액 | 500억 원 이상 (`hts_avls` × 1억) | 유동성 확보, 소형주 제외 |
| 거래량 | 20일 평균 거래량 10만주 이상 (일봉 기반) | 체결 가능성 확보 (장 전에도 정확) |
| 블랙리스트 | `watchlist.yaml` 블랙리스트 제외 | 위험 종목 차단 |

> **주의**: `hts_avls`는 억 원 단위, `acml_vol`은 장 전 거의 0이므로 일봉 20일 평균을 사용한다. (2026-03-30 수정)

**2차 필터: 기술적 조건** (API 호출 필요)
→ 결과: 300개 → 최종 30~50개 후보 종목 → 전략 엔진 전달

| 필터 | 조건 | 목적 |
|------|------|------|
| MA 정배열 | 5일 > 20일 > 60일 이동평균 | 상승 추세 종목 선별 |
| RSI | 30 < RSI(14) < 70 | 과매수/과매도 제외 |
| 수급 | 외국인/기관 최근 3일 순매수 | 스마트머니 동향 확인 |
| 변동성 | ATR(14) 적정 범위 | 극단 변동 종목 제외 |

### 4.3 보유 종목 필수 감시 (고아 종목 방지)

**핵심 규칙: 보유 중인 종목은 스크리닝 결과와 무관하게 항상 활성 감시 목록에 포함된다.** 사놓고 신호 평가를 안 하는 종목이 절대 생기지 않는다.

#### 4.3.1 활성 감시 목록 구성 (매일 장 시작 전)

매일 다음 3가지를 합쳐서 오늘의 활성 감시 목록을 자동 구성한다:

1. **보유 종목 전체** (무조건 포함 — 절대 제외 불가)
2. **고정 유니버스** (ETF + 수동 등록 관심종목)
3. **오늘의 스크리닝 후보군** (30~50개)

→ 중복 제거 후 전략 엔진이 이 목록 전체에 대해 신호를 평가한다.

#### 4.3.2 보유 종목이 스크리닝에서 빠졌을 때

보유 중인 종목이 스크리닝에서 제외되는 것은 기술적 조건 악화 신호이다. 감시는 계속하되, 매도 신호를 점진적으로 강화한다:

| 상태 | 조건 | 시스템 액션 |
|------|------|-------------|
| 탈락 1일차 | 오늘 스크리닝에서 제외됨 | 주의 태그 + Discord 알림 + 감시 유지 |
| 탈락 3일 연속 | 3일 연속 제외 | 매도 신호 강화 (Confidence +0.2 가중) |
| 탈락 + 손실구간 | 스크리닝 탈락 + 현재 -2% 이하 | 손절매 기준 강화 (-3%→-2%) → 조기 청산 유도 |
| 탈락 + 수익구간 | 스크리닝 탈락 + 현재 +3% 이상 | 익절매 검토 알림 (조건 악화 전 수익 확정) |

#### 4.3.3 고아 종목 방지 일일 체크 (15:40)

매일 장 마감 후 다음 항목을 자동 체크한다:

- **보유 기간 체크**: 5거래일 이상 보유 + 매도 신호 한 번도 없었던 종목 → Discord 알림
- **전략 미적용 체크**: 보유 중인데 어떤 전략에도 포함 안 된 종목 → 즉시 알림 + 수동 판단 요청
- **수익률 정체**: 10거래일+ 보유 + 수익률 ±1% 이내 → 청산 검토 권고
- **최대 보유 기간**: 20거래일(약 1개월) 초과 시 AI에게 종합 평가 요청 → Discord 보고

### 4.4 종목 선정 흐름도

```
[1] 전일 데이터로 1차 필터 (2,500개 → 300개)
[2] API로 2차 기술적 필터 (300개 → 50개)
[3] 보유 종목 + 고정 ETF 합치 → 오늘의 활성 감시 목록
[4] 전략 엔진이 활성 목록 전체에 대해 신호 평가
[5] 매일 15:40 고아 종목 방지 체크
```

이 구조로 보유 중인 종목은 절대 감시에서 빠지지 않으며, 스크리닝에서 탈락한 보유 종목에 대해서는 오히려 매도 신호를 강화하여 적절한 청산을 유도한다.

## 5. 전략 엔진

### 5.1 장세 판단 모듈 (market_regime.py)

모든 전략을 적용하기 전에 현재 시장 상태를 판단한다. 장세에 따라 활성화할 전략 조합이 달라진다.

| 장세 구분 | 판단 기준 | 활성화 전략 예시 | 포지션 비중 |
|-----------|-----------|-----------------|-------------|
| 강한 상승장 | KOSPI 20일 MA > 40일 MA, ADR > 110% | 추세추종, 변동성 돌파 | 최대 80% |
| 약한 상승장 | KOSPI 5일 MA > 20일 MA, ADR 90~110% | 이동평균선, RSI 역발 | 최대 60% |
| 횡보합 (박스권) | KOSPI 변동폭 < 2%, 거래량 감소 | 엔벨로프 역발, 레인지 트레이딩 | 최대 40% |
| 약한 하락장 | KOSPI 5일 MA < 20일 MA, ADR < 90% | 뉴스 감성 기반 단타, 방어적 종목 | 최대 30% |
| 강한 하락장 | KOSPI 20일 MA < 40일 MA, VIX 급등 | 현금 비중 확대, 전략 정지 | 최대 10% |

### 5.2 전략 모듈 설계

모든 전략은 BaseStrategy 추상 클래스를 상속받아 통일된 인터페이스로 구현된다.

#### 5.2.1 BaseStrategy 인터페이스

- `generate_signal(data)` → BUY / SELL / HOLD 신호 반환
- `get_parameters()` → 현재 파라미터 반환
- `update_parameters(params)` → 파라미터 업데이트
- `get_confidence()` → 신호 확신도 (0.0 ~ 1.0) 반환
- `backtest(historical_data)` → 백테스트 결과 반환

#### 5.2.2 전략 목록 (모두 일봉 기반)

모든 전략은 **일봉(Daily)** 데이터를 기반으로 신호를 생성한다. 장중에는 일봉 + 당일 현재가 바(시가/고가/저가/현재가)를 합쳐서 전략에 전달한다.

| 전략명 | 유형 | 적합 장세 | 핵심 지표 | 매수 조건 | 원저자/출처 |
|--------|------|-----------|-----------|-----------|-------------|
| 변동성 돌파 | 추세추종 | 상승장 | Range × K, 거래량 | 전일 변동폭의 K배 돌파 + 거래량 1.5배 | Larry Williams (1999) |
| 이동평균선 교차 | 추세추종 | 상승/횡보 | MA5, MA20, MA60 (일봉) | 정배열(5>20>60) + 골든크로스 | Granville (1960s) |
| RSI 역발 | 역발 | 횡보/하락 | RSI(14일), BB(20,2) | RSI ≤ 30 + 볼린저 하단 터치 | Wilder (1978), Bollinger (2001) |
| 엔벨로프 | 역발 | 횡보 | MA20 ±5% (일봉) | 하단밴드 터치 후 반등 | 고전적 기술분석 |
| 뉴스 감성 | 이벤트 | 전체 | AI 감성점수 + MA20 (일봉) | 감성 > 0.3 + 가격 > MA20 | AI 기반 복합전략 |

### 5.3 전략 조합 및 신호 합산

장세 판단 결과에 따라 복수의 전략이 동시에 활성화되며, 각 전략의 신호를 가중평균하여 최종 매매 결정을 내린다.

**합산 공식**: `Final_Score = Σ(Strategy_Signal × Weight × Confidence)`

- 각 전략의 Weight는 최근 성과에 따라 동적으로 조정
- Confidence는 해당 전략의 현재 시장 적합도
- **Final_Score > 0.6 → 매수**, **Final_Score < -0.6 → 매도**
- 0.4 ~ 0.6 범위는 주의 관찰 → 알림만 발송

### 5.4 자산별 리밸런싱 전략

투입된 현금 총액을 기준으로, 장세 판단 결과에 따라 자산군별 목표 비중을 자동 조정한다.

| 장세 | 주식 | 금/은 | 채권 | 현금 | 기타 (달러/원유) |
|------|------|-------|------|------|-----------------|

#### 5.4.3 리밸런싱 실행 규칙

리밸런싱은 정기 주기 + 긴급 트리거 혼합 방식으로 실행된다:

- **정기 리밸런싱**: 매주 금요일 장 마감 후 16:30에 현재 비중과 목표 비중을 비교하여 조정
- **긴급 리밸런싱**: 장세 전환 감지 시(예: 상승장 → 하락장) 즉시 목표 비중 변경 및 조정
- **허용 편차 범위**: 목표 비중 대비 ±5% 이내면 리밸런싱 생략 (불필요한 거래 비용 방지)
- **비중 이탈 트리거**: 특정 자산군이 목표 대비 ±10% 이상 벗어나면 요일과 무관하게 즉시 리밸런싱
- **최소 거래 금액**: 리밸런싱 매매 금액이 5만원 미만이면 생략
- **점진적 이동**: 목표와의 차이의 50%씩만 조정 (급격한 포트폴리오 변화 방지, 나머지는 다음 리밸런싱 시)

#### 5.4.4 리밸런싱 계산 예시

투입 현금 1,000만원, 횡보합장 예시:

| 자산군 | 목표 비중 | 목표 금액 | 현재 금액 | 조정 행동 |
|--------|-----------|-----------|-----------|-----------|
| 주식 | 25% | 250만원 | 350만원 | 100만원 매도 |
| 금/은 ETF | 20% | 200만원 | 150만원 | 50만원 매수 |
| 채권 ETF | 20% | 200만원 | 200만원 | 유지 (편차 0%) |
| 현금 | 30% | 300만원 | 250만원 | 매도 대금으로 충당 |
| 기타 | 5% | 50만원 | 50만원 | 유지 |

주식을 100만원어치 매도하고, 그 대금으로 금/은 ETF 50만원 매수 + 현금 50만원 확보하는 식으로 작동한다. 모든 계산은 투입된 현금 총액을 기준으로 하며, 평가손익은 반영하지 않는다.

## 6. 리스크 관리

### 6.1 포지션 사이징

각 거래의 포지션 크기를 결정하는 규칙을 정의한다.

| 규칙 | 값 | 설명 |
|------|-----|------|
| 최대 단일 종목 비중 | 총자산의 10% | 하나의 종목에 과도한 집중 방지 |
| 단일 거래 위험금액 | 총자산의 2% | 한 번의 거래에서 최대 손실 한도 |
| 일일 최대 손실 | 총자산의 5% | 하루 총 손실 한도 초과 시 거래 중지 |
| 총 포지션 수 | 최대 10종목 | 관리 가능한 범위 유지 |
| 현금 비중 최소 | 20% | 긴급 상황 대비 유동성 확보 |

### 6.2 손절매 / 익절매

| 타입 | 기본값 | 조건 |
|------|--------|------|
| 손절매 (Stop-Loss) | -3% | 매수가 대비 3% 하락 시 자동 매도 |
| 추적 손절매 (Trailing Stop) | -2% | 최고점 대비 2% 하락 시 매도 (수익구간) |
| 익절매 (부분) | +5% | 목표가 도달 시 50% 부분 매도 |
| 익절매 (전체) | +10% | 목표가 도달 시 전량 매도 |
| 시간 기반 손절 | 3일 | 3일 이내 수익전환 없으면 재평가 |

### 6.3 서킷 브레이커 (비상 정지)

시스템 보호를 위한 자동 정지 조건을 설정한다.

- 일일 총 손실이 5%를 초과하면 당일 모든 거래 중지
- 연속 3회 손절매 발생 시 해당 전략 일시 정지 (24시간)
- 주간 손실이 10%를 초과하면 전체 시스템 정지 및 수동 확인 필요
- API 오류 연속 5회 발생 시 자동 정지 및 알림

## 7. 주문 실행 모듈

### 7.1 주문 흐름

1. **신호 생성**: 전략 엔진에서 BUY/SELL 신호 발생
2. **리스크 체크**: 포지션 사이징, 손절매 설정, 비중 한도 확인
3. **주문 생성**: 시장가 또는 지정가 주문 생성
4. **API 호출**: 한국투자증권 API로 주문 전송
5. **체결 확인**: 웹소켓으로 체결 결과 수신
6. **결과 기록**: DB에 거래 내역 저장 + 알림 발송

### 7.2 주문 유형

| 상황 | 주문 방식 | 이유 |
|------|-----------|------|
| 신호 확신도 높음 (>0.8) | 시장가 매수 | 빠른 체결 우선 |
| 신호 확신도 중간 (0.6~0.8) | 지정가 매수 (현재가 -0.3%) | 슬리피지 최소화 |
| 손절매 발동 | 시장가 매도 | 즉시 처리 필수 |
| 익절매 | 지정가 매도 | 목표가에서 정확히 체결 |

### 7.3 슬리피지 대응

실제 체결가와 예상 가격의 차이(슬리피지)를 모니터링하고, 전략 평가 시 반영한다.

- 체결 후 실제 체결가와 신호 발생 시점 가격 비교 기록
- 평균 슬리피지가 0.5% 초과 시 주문 방식 조정 검토
- 백테스트 결과에 슬리피지 비용 0.1~0.3% 반영

## 8. 기록 및 평가 시스템

### 8.1 일일 기록 (매일 장 마감 후 15:40)

매 거래일 장 마감 후 자동으로 일일 보고서를 생성한다.

- 당일 전체 거래 내역 (매수/매도 종목, 수량, 가격)
- 각 전략별 발생한 신호 및 실행 결과
- 포트폴리오 수익률 (실현손익 + 평가손익)
- 장세 판단 결과 및 활성화된 전략 목록
- 뉴스 감성 점수 요약
- 특이 사항 (서킷브레이커 발동, API 오류 등)

### 8.2 주간 전략 평가 (매주 금요일 16:00)

매 주 금요일에 자동으로 전략 평가 보고서를 생성하고, AI를 활용하여 개선안을 도출한다.

#### 8.2.1 평가 지표

| 지표 | 목표 | 설명 |
|------|------|------|
| 주간 수익률 | > 0% | 해당 주의 총 수익률 |
| 실현손익 수익률 | > 0% | 실제 체결된 거래의 수익률 |
| Sharpe Ratio | > 1.0 | 위험 대비 수익 효율성 |
| 최대 드로다운 | < 5% | 최고점 대비 최대 하락폭 |
| 승률 | > 55% | 전체 거래 중 수익 거래 비율 |
| Profit Factor | > 1.5 | 총 수익 / 총 손실 비율 |
| 전략별 승률 | - | 각 전략의 개별 성과 비교 |

#### 8.2.2 자동 개선 프로세스

1. 주간 성과 데이터 수집 (전략별 승률, 수익률, 손익비)
2. AI API에 성과 데이터 + 시장 환경 전달하여 분석 요청
3. AI가 파라미터 조정안 제안 (예: RSI 기간 14→12, 손절매 -3%→-2.5%)
4. 제안된 파라미터로 백테스트 수행 (최근 3개월 데이터)
5. 백테스트 결과가 개선되면 파라미터 업데이트, 아니면 유지
6. 변경 이력 버전 관리 (strategies.yaml에 버전 기록)

> **중요**: 자동 개선은 제안 → 백테스트 → 검증 → 적용의 안전한 파이프라인을 따른다. 검증 없이 자동으로 파라미터를 변경하지 않는다.

## 12. 백테스팅 엔진

### 12.1 백테스팅 원칙

- 최소 3년 이상의 과거 데이터로 테스트 (상승장, 하락장, 횡보합 모두 포함)
- 거래 비용 반영: 수수료 (0.015%) + 세금 (0.18%) + 슬리피지 (0.1~0.3%)
- Walk-Forward 분석: 80% 학습 데이터 + 20% 검증 데이터
- 오버피팅 방지: 단순한 파라미터, Out-of-Sample 테스트 필수
- 모의투자 계좌로 최소 1개월 페이퍼 트레이딩 후 실전 전환

### 12.2 평가 기준

백테스트 결과 다음 기준을 만족해야 실전 배포한다.

| 지표 | 최소 기준 | 권장 기준 |
|------|-----------|-----------|
| Sharpe Ratio | > 1.0 | > 1.5 |
| 최대 드로다운 | < 20% | < 10% |
| Profit Factor | > 1.2 | > 1.5 |
| 승률 | > 50% | > 55% |
| Out-of-Sample 성과 | In-Sample의 70% 이상 | In-Sample의 85% 이상 |

## 14. 추가 고려사항 (조사 결과 반영)

### 14.1 구상에 추가할 요소들

인터넷 조사 결과, 원래 구상에 더해 다음 요소들을 추가하는 것을 권장한다.

#### 14.1.1 모의투자 우선 개발

한국투자증권은 모의투자 환경을 제공한다. 실전 자금을 투입하기 전에 반드시 모의투자로 충분히 테스트해야 한다. 모의투자 API 호출 제한이 초당 5건으로 더 빡빡하므로, 개발 단계에서는 호출 간격을 0.25초 이상으로 설정해야 한다.

#### 14.1.2 투자자별 매매동향 활용

외국인/기관 순매수 데이터는 상승장에서 특히 유용한 보조 지표이다. 외국인이 3일 연속 순매수하는 종목은 추세 전략의 확신도를 높이는 데 활용할 수 있다.

#### 14.1.3 화이트리스트 및 블랙리스트

종목 필터링을 위해 화이트리스트(거래 대상 종목)와 블랙리스트(절대 거래 금지 종목)를 설정한다. 관리종목, 거래량 초저 종목, 정리매매 대상 종목 등을 자동으로 블랙리스트에 추가한다.

#### 14.1.4 보안 강화

API 키와 시크릿은 환경변수 또는 .env 파일로 관리하고, Git에는 절대 커밋하지 않는다. settings.yaml의 민감한 정보는 .env에서 참조하는 구조로 설계한다.

#### 14.1.5 장애 복구 및 로깅

모든 API 호출과 거래를 상세히 로깅한다. WebSocket 연결 끊김 시 자동 재연결, API 오류 시 지수 백오프 재시도 등의 장애 복구 로직을 포함한다. 로그는 일별로 파일 회전하여 30일치를 보관한다.

#### 14.1.6 휴장일 캘린더 관리

시스템은 매일 기동 시 휴장일 여부를 확인하고, 휴장일이면 모든 매매 기능을 비활성화한다.

- **데이터 소스**: KRX 휴장일 조회 API 또는 로컬 JSON 캘린더 파일 병행 사용
- **임시공휴일/대체공휴일 처리**: 매년 초에 해당 연도 휴장일 목록을 수동으로 확인하고 JSON 파일 업데이트
- **휴장일에도 뉴스 수집은 정상 수행** (다음 거래일 장세 판단에 활용)
- 연말연시/추석 등 장기 휴장 기간에는 시스템 정상 가동을 Discord로 알림

#### 14.1.7 배당락일/권리락일 갭 처리

배당락일에는 배당금만큼 주가가 하락하는 갭이 발생한다. 이를 일반 하락과 혼동하면 불필요한 손절매가 발동될 수 있다.

- 대응: KRX 배당락일/권리락일 캘린더를 DB에 저장하여 사전 확인
- 대응: 배당락일 해당 종목은 당일 손절매 판단에서 배당금 상당액을 차감하여 평가
- 대응: 배당락일 전일 매수 신호는 보류 처리 (락일 갭으로 즉시 손실 발생 방지)

#### 14.1.8 동시호가 구간 처리 (08:30~09:00)

장 시작 전 동시호가 구간(08:30~09:00)에서는 주문은 가능하지만 체결은 09:00에 일괄 처리된다. 이 구간의 특성을 이해하고 적절히 처리해야 한다.

- 대응: 08:30~09:05 구간에는 시스템이 신규 매수/매도 주문을 발송하지 않음
- 대응: 장 시작 직후 09:00~09:05 시초 변동성 구간도 관망만 수행
- 대응: 실제 매매 시작은 09:05부터 (시초 변동성 회피)
- 대응: 장 마감 동시호가(15:20~15:30)에도 동일하게 신규 주문 발송 중지

## 15. ChatGPT Codex 5.3 LLM 연동

### 15.1 연동 방식 및 비용 구조

ChatGPT Codex 5.3를 OAuth 방식으로 연동하여, ChatGPT Plus 구독료($20/월) 외에는 추가 비용이 발생하지 않는 구조로 설계한다. Codex CLI를 통해 ChatGPT 계정으로 로그인하면 OAuth 토큰이 자동 발급되며, 별도 API 키 설정 없이 사용 가능하다.

| 항목 | 상세 |
|------|------|
| 인증 방식 | ChatGPT OAuth (Sign in with ChatGPT) |
| 사용 모델 | GPT-5.3-Codex (비용 효율적, 구독 범위 내) |
| 월 비용 | ChatGPT Plus $20/월 (추가 비용 없음) |
| 호출 방식 | Python openai 라이브러리로 API 호출 |
| 토큰 관리 | OAuth 토큰 자동 갱신 (24시간 주기) |
| 사용량 제한 | ChatGPT Plus 플랜 사용량 한도 내 |

### 15.2 LLM 활용 영역

ChatGPT Codex 5.3는 다음 영역에서 자동으로 활용된다. 이 영역들은 사람의 개입 없이 자동으로 실행되며, 결과를 Discord로 보고한다.

| 활용 영역 | 시점 | 입력 데이터 | 출력 |
|-----------|------|-------------|------|
| 뉴스 감성 분석 | 매일 3회 (08:00/12:00/14:00) | 뉴스 본문 + 관련 종목 | 감성 점수 (-1.0 ~ +1.0) |
| 일일 투자 평가 | 매일 15:40 (장 마감 후) | 당일 거래내역, 수익률, 신호 이력 | 오늘의 판단 평가 보고서 |
| 주간 전략 평가 | 매주 금요일 16:00 | 주간 성과 + 시장환경 + 전략별 통계 | 전략 개선안 + 파라미터 조정 제안 |
| 전략 업그레이드 | 주간 평가 후 자동 | AI 제안 + 백테스트 결과 | 검증된 파라미터 업데이트 |
| 사용자 지시 응답 | Discord !ask 명령어 | 사용자 질문 + 현재 포트폴리오 | AI 분석 결과 응답 |

### 15.3 일일 평가 보고 프로세스 (자동)

매일 장 마감 후 15:40에 자동으로 실행된다:

1. 당일 거래 데이터, 신호 이력, 수익률 수집
2. ChatGPT Codex에 데이터 전달 + 평가 요청 프롬프트 전송
3. AI가 오늘의 판단 평가, 잘한 점/못한 점, 개선 제안 생성
4. 결과를 DB 저장 + Discord #일일-보고 채널에 발송

### 15.4 주간 전략 업그레이드 프로세스 (자동)

매주 금요일 16:00에 자동으로 실행된다:

1. 주간 성과 데이터 수집 (전략별 승률, 수익률, 손익비, Sharpe)
2. ChatGPT Codex에 성과 데이터 + 시장 환경 전달 → 분석 및 개선안 요청
3. AI가 파라미터 조정안 제안 (예: RSI 기간 14→12, 손절매 -3%→-2.5%)
4. 제안된 파라미터로 백테스트 자동 수행 (최근 3개월 데이터)
5. 백테스트 결과가 개선되면 파라미터 업데이트, 아니면 유지
6. 결과를 DB 저장 + Discord #주간-보고 채널에 발송

> **중요**: 모든 자동 개선은 반드시 백테스트 검증을 거친 후에만 적용된다. Discord에서 `!upgrade` 명령으로 수동 트리거도 가능하다.

### 15.5 사용자 지시 모드

Discord #명령-제어 채널에서 `!ask` 명령어로 직접 AI에게 질문하거나 지시할 수 있다. 예시:

- `!ask 오늘 삼성전자 매도 판단이 적절했나?`
- `!ask 현재 장세에서 엔벨로프 전략 파라미터를 어떻게 조정하면 좋겠나?`
- `!evaluate rsi_envelope` → 특정 전략 즉시 평가 실행

## 16. 로컬 대시보드 UI (Streamlit)

### 16.1 개요

Ubuntu 노트북의 브라우저에서 시스템 상태와 성과를 시각적으로 모니터링할 수 있는 대시보드를 Streamlit으로 구축한다. Streamlit은 무료 오픈소스이며, `pip install streamlit`으로 설치한다. `localhost:8501`에서 접속한다.

### 16.2 대시보드 페이지 구성

| 페이지 | 표시 내용 | 갱신 주기 |
|--------|-----------|-----------|
| 메인 대시보드 | 총 자산, 오늘 수익률, 활성 전략 상태, 보유종목 현황 | 실시간 (5초) |
| 거래 내역 | 오늘/주간/월간 거래 내역 테이블, 종목별 손익 | 실시간 |
| 전략 모니터 | 전략별 승률, 수익률, 신호 이력 차트 | 매 5분 |
| 수익률 차트 | 누적 수익률 그래프, 드로다운 차트, KOSPI 대비 | 매일 |
| 뉴스 피드 | 최근 뉴스 + 감성 점수, 종목 매핑 | 뉴스 수집 시 |
| AI 분석 로그 | 일일/주간 AI 평가 보고서 이력 | 일일/주간 |
| 설정 페이지 | 관심종목 관리, 전략 파라미터 조회/수정 | 수동 |

### 16.3 기술 스택

- **Streamlit**: Python 웹 대시보드 프레임워크 (무료)
- **Plotly**: 인터랙티브 차트 라이브러리 (무료)
- **SQLite 직접 연결**: 별도 백엔드 서버 불필요
- **자동 시작**: systemd 또는 crontab으로 시스템 기동 시 함께 시작

## 17. Git 및 배포 워크플로

### 17.1 개발 환경

개발은 GitHub 레포지토리에서 진행하고, Claude Code를 활용하여 코드 작성/수정을 한 다음, 로컬 Ubuntu 노트북에 git clone으로 가져와서 실행한다.

#### 17.1.1 개발 → 배포 흐름

1. Claude Code에서 코드 작성/수정
2. GitHub에 push
3. 로컬 노트북에서 `git pull` 또는 초기 `git clone`
4. `pip install -r requirements.txt` 로 의존성 설치
5. `.env` 파일에 API 키/시크릿 설정 (최초 1회만)
6. `systemctl restart auto-trader` 로 시스템 재시작

#### 17.1.2 레포지토리 구조

```
auto-trader/
├── .gitignore                 # .env, data/db/, data/logs/ 제외
├── .env.example               # 환경변수 템플릿 (실제 값 비워둘)
├── requirements.txt           # 의존성 목록
├── setup.sh                   # 초기 설치 스크립트
├── auto-trader.service         # systemd 서비스 파일
└── (... 소스코드 ...)
```

### 17.2 초기 설치 스크립트 (setup.sh)

로컬 노트북에서 최초 1회 실행하는 설치 스크립트:

1. Python 가상환경 생성 및 의존성 설치
2. .env 파일 생성 가이드 (사용자가 키 입력)
3. SQLite DB 초기화
4. systemd 서비스 등록 (부팅 시 자동 시작)
5. Discord Bot 토큰 설정
6. ChatGPT Codex OAuth 로그인

### 17.3 업데이트 방법

코드 수정 후 배포:

```bash
cd ~/auto-trader && git pull origin main
pip install -r requirements.txt  # (의존성 변경 시)
sudo systemctl restart auto-trader
```

DB 스키마 변경 시에는 마이그레이션 스크립트를 포함하여 기존 데이터를 보존한다.

## 19. 자동매매 봇 개발 시 흔한 문제점 및 대응

인터넷 조사 결과, 자동매매 봇 개발에서 반복적으로 발생하는 문제들을 정리하고, 본 시스템의 대응 상태를 함께 기록한다.

### 19.1 이미 대응된 문제점

| 문제 | 설명 | 대응 위치 |
|------|------|-----------|
| 과적합(Overfitting) | 백테스트에서만 좋고 실전에서 실패하는 전략 | 11장: Walk-Forward, Out-of-Sample 테스트 |
| 슬리피지 무시 | 체결가 차이로 수익률 50% 이상 감소 가능 | 6.3절: 슬리피지 모니터링 및 백테스트 반영 |
| 리스크 관리 부재 | 손절매 없이 소액 손실이 대형 손실로 확대 | 5장: 손절매/익절매/서킷브레이커 |
| API 키 보안 취약 | 키 탈취로 자산 손실 사례 증가 | 13.1.4절: .env 파일, .gitignore 관리 |
| 장애 복구 부재 | API 오류/서버 장애 시 무방비 상태 | 13.1.5절: 자동 재연결, 지수 백오프 |
| 장세 변화 미대응 | 추세장 전략이 횡보합장에서 실패 | 4.1절: 장세 판단 모듈 + 전략 조합 |
| 고아 포지션 | 매수 후 모니터링 누락으로 방치 | 4.6절: 보유종목 보호 로직 |
| 거래비용 과다 | 빈번한 매매로 수수료가 수익 초과 | 4.4.3절: 최소거래 5만원, 편차 ±5% 허용 |

### 19.2 추가 대응 필요 문제점 (신규)

#### 19.2.1 백테스트 vs 실전 환경 불일치

백테스트에서는 신호 발생 즉시 체결을 가정하지만, 실전에서는 API 호출 지연, 미체결, 부분 체결 등이 발생한다.

- 대응: 백테스트에 체결 지연(1분봉 단위) 및 부분체결 시뮬레이션 추가
- 대응: 모의투자 1개월 후, 백테스트 결과와 모의투자 결과를 비교하여 캘리브레이션
- 대응: 실전 전환 후에도 주간 보고서에 백테스트 vs 실전 성과 비교 항목 포함

#### 19.2.2 중복 주문 방지 (멱등성 보장)

네트워크 장애나 API 타임아웃 시 주문이 중복 전송될 수 있다.

- 대응: 주문마다 고유 ID 부여 + DB에 주문 상태 기록
- 대응: 주문 전송 전 최근 5분 내 동일 종목/방향/수량 주문 존재 여부 확인
- 대응: 주문 전송 → 체결 확인 사이 잠금(Lock) 처리

#### 19.2.3 생존편향(Survivorship Bias) 없는 데이터

현재 상장된 종목만으로 백테스트하면 과거 상장폐지 종목이 제외되어 성과가 과장된다.

- 대응: 백테스트 데이터에 과거 상장폐지/관리종목 포함 (KRX 과거 데이터 활용)
- 대응: ETF/지수 위주 전략은 생존편향 영향이 적으므로 초기에는 ETF 중심으로 시작

#### 19.2.4 봇 수동 개입 충동 방지

연속 손실 시 봇을 멈추거나, 수익 시 비중을 높이는 감정적 개입이 자동화의 의미를 훼손한다.

- 대응: 운영 규칙 문서화 — 최소 1개월은 개입 없이 관찰
- 대응: 수동 정지/재개 시 Discord에 사유 기록 필수화
- 대응: 주간 보고서에 수동 개입 횟수 및 영향 추적

#### 19.2.5 극단적 변동성 상황 (Gap, 서킷브레이커)

장 시작 시 급상승/급하락(Gap)이나, 거래소 서킷브레이커 발동 시 손절매가 예상가격에 체결되지 못한다.

- 대응: 장 시작 전 08:50에 전일 종가 대비 Gap 비율 체크
- 대응: Gap이 ±3% 이상이면 해당 종목 신규 매수 보류 (30분 관찰 후 판단)
- 대응: 거래소 서킷브레이커 발동 시 모든 신규 주문 즉시 정지, 해제 후 재평가

## 20. 자기 진화형 전략 개선 (HyperAgent 개념 적용)

### 20.1 배경: 메타의 HyperAgents

메타가 2026년 3월 발표한 HyperAgents는 AI가 단순히 주어진 문제를 해결하는 것을 넘어, 자신의 학습 방식과 개선 절차 자체를 수정하며 성능을 높이는 아키텍처이다. 핵심은 계획–실행–검증의 반복 루프로, 과제 수행 능력뿐 아니라 더 잘 배우는 능력까지 동시에 진화한다.

> 참고: 박찬. (2026, 3월 25일). 메타, 자가 진화형 AI '하이퍼에이전트' 발표. AI타임스.

### 20.2 전체 자기 진화 대상 영역

자동매매 시스템의 모든 구성 요소에 자기 진화를 적용한다. 각 영역은 변경 시 위험도가 다르므로, 안전한 것부터 순서대로 학습한다.

| 영역 | 진화 대상 예시 | 위험도 | 학습 단계 |
|------|---------------|--------|-----------|
| 전략 파라미터 | RSI 기간, MA 일수, K값 등 수치 조정 | 낮음 | 1단계 (1~2개월차) |
| 매수/매도 신호 합산 | 신호 합산 공식, 임계값 (0.6) 조정 | 낮음 | 1단계 |
| 전략 선택 가중치 | 장세별 전략 조합 비중 조정 | 중간 | 2단계 (3~4개월차) |
| 종목 스크리닝 기준 | 시가총액/거래량 필터, 기술적 조건 조정 | 중간 | 2단계 |
| 손절매/익절매 기준 | 종목/전략별 최적 손절 수준 학습 | 높음 | 3단계 (5~6개월차) |
| 리밸런싱 비중/주기 | 장세별 목표 비중 테이블 자체 진화 | 높음 | 4단계 (6개월 이후) |
| 뉴스 감성 활용법 | 감성 점수 → 매매 반영 가중치 조정 | 높음 | 4단계 |
| 평가 기준/분석 프레임워크 | 승률 vs Profit Factor 중 어떤 기준을 중시할지 | 매우 높음 | 4단계 (실험적) |

### 20.3 계층적 순차 학습 방법론

여러 영역을 한꺼번에 학습시키면, 성과 변화의 원인을 특정할 수 없다. 반드시 하나씩 순서대로 학습하고 검증한 후 고정하고 다음으로 넘어간다.

#### 20.3.1 데이터 분리 규칙 (Train / Validation / Test)

과거 데이터(5년 권장)를 3개 구간으로 분리하여 모든 단계에 동일하게 적용한다.

| 구간 | 기간 예시 | 비중 | 용도 |
|------|-----------|------|------|
| Train (학습용) | 2021년 ~ 2023년 | 60% | 1~4단계 순차 학습에 사용 |
| Validation (검증용) | 2024년 | 20% | 각 단계의 결과 확인용 |
| Test (최종 테스트) | 2025년 | 20% | 모든 단계 완료 후 딱 1회 검증 |

> **핵심**: 1~4단계 모두 동일한 Train 데이터로 학습한다. 동일한 데이터 위에서 순차적으로 쌓아올려야 각 단계의 개선 효과를 정확히 측정할 수 있다.

#### 20.3.2 순차 학습 프로세스

1. **1단계**: Train 데이터로 전략 파라미터 + 신호 합산 최적화 → Validation으로 확인 → 확정 후 고정
2. **2단계**: 1단계 고정 상태에서, 같은 Train 데이터로 전략 가중치 + 스크리닝 기준 학습 → Validation 확인 → 고정
3. **3단계**: 1~2단계 고정 상태에서, 손절매/익절매 기준 학습 → Validation 확인 → 고정
4. **4단계**: 1~3단계 고정 상태에서, 리밸런싱 비중 + 뉴스 활용법 + 평가 프레임워크 학습
5. **최종**: 모든 단계 완료 후, Test 데이터로 딱 1회 최종 검증
   - Test 통과 시 모의투자 전환, 실패 시 문제 단계로 돌아가 재검토

### 20.4 시장 국면 분류 및 데이터 수집

Train 데이터 안에 다양한 시장 국면이 골고루 포함되어야 한다. 특정 국면만 있는 데이터로 학습하면 그 국면에서만 작동하는 과적합이 발생한다.

#### 20.4.1 시장 국면 8가지 분류

| 국면 | 판단 기준 | KOSPI 예시 기간 | 핵심 특징 |
|------|-----------|----------------|-----------|
| 강한 상승장 (Bull) | 20MA > 60MA, 월간 +5% 이상 | 2021.01~2021.06 | 높은 신뢰도로 추세추종 유리 |
| 약한 상승장 (Mild Bull) | 5MA > 20MA, 월간 +1~5% | 2023.01~2023.06 | 느릿한 상승, 선별적 접근 필요 |
| 횡보합 (Range) | 변동폭 <2%, 거래량 감소 | 2023.07~2023.12 | 박스권 매매, 역발 전략 적합 |
| 조정구간 (Correction) | 고점 대비 -10~20% 하락 | 2021.07~2021.12 | 단기 하락, 빠른 반등 가능성 |
| 강한 하락장 (Bear) | 20MA < 60MA, 월간 -5% 이상 | 2022.01~2022.09 | 현금 비중 확대, 방어적 운영 |
| 급락/패닉장 (Crash) | 주간 -10% 이상 급락 | 2020.02~2020.03 | 서킷브레이커 발동, 거래 중지 필요 |
| 회복기 (Recovery) | 저점에서 반등, 5MA > 20MA 전환 | 2020.04~2020.12 | 공격적 매수 기회 |
| 고변동성 횡보합 | 일간 변동폭 >2%, 방향성 없음 | 2022.10~2022.12 | 위험 관리 강화 필요 |

#### 20.4.2 데이터 수집 및 구간 자동 분류 로직

과거 KOSPI 일봉 데이터를 수집한 후, 다음 로직으로 자동 구간 분류한다:

1. 월간 수익률 계산 (KOSPI 월말 종가 기준)
2. 20일/60일 이동평균선 배열 상태 판단
3. 일간 변동폭 평균, 거래량 추이 분석
4. 위 기준으로 8가지 국면 중 하나로 자동 태그
5. 각 구간의 시작일/종료일/기간/국면을 DB에 저장

> Train 데이터 안에 최소 5가지 이상의 국면이 포함되어야 한다. 특정 구간이 부족하면 데이터 수집 기간을 늘려야 한다.

### 20.5 안전장치

- **모든 변경은 백테스트 검증 후에만 적용** (모든 단계 공통)
- **한 번에 하나의 영역만 변경**: 변경 → Validation 검증 → 확정 → 다음 영역
- **2단계 이상 변경은 Discord 보고 + 사용자 승인 필수**
- **4단계 변경은 최소 1개월 A/B 테스트 후 채택**
- **모든 변경 이력 strategy_params 테이블에 버전 관리, 즉시 롤백 가능**
- **Validation에서 성과가 떨어지면 변경 취소하고 원래 설정 유지**
